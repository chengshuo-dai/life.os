# Life.OS Technical Integration Manifest (V1)

This document outlines the required API endpoints and data structures to transform the **Life.OS** frontend designs into a functional, persistent application.

## 1. Authentication & Session Management
The entry gateway (`SCREEN_4`, `SCREEN_31`) currently simulates a boot sequence. For a functional MVP, these interactions must map to an auth provider.

- **Endpoint**: `POST /api/v1/auth/initialize`
- **Frontend Trigger**: `IDENTIFY_USER` interaction on the Gateway.
- **Data Payload**:
  ```json
  {
    "biometric_signature": "string (hashed)",
    "device_metadata": "object"
  }
  ```
- **Response**: JWT or Session Token, User Profile (Credits, Unlocked Nodes).

## 2. Narrative Engine (Main Console)
The Narrative Hub (`SCREEN_5`, `SCREEN_27`) needs to dynamically populate cards and track progress.

- **Endpoint**: `GET /api/v1/narratives`
- **Data Structure**:
  ```json
  [
    {
      "id": "NARRATIVE_082_MED",
      "status": "UNLOCKED | LOCKED | COMPLETED",
      "progress": 45,
      "metadata": { "recovery_level": "HIGH" }
    }
  ]
  ```
- **Action Endpoint**: `POST /api/v1/narratives/{id}/choice`
  - Triggered by user decisions (e.g., "Defibrillate" vs "Wait").
  - Maps to the branching logic in `SCREEN_18`.

## 3. Credit System & Paywalls
The "Curiosity Paywall" (`SCREEN_7`) requires a secure transaction flow.

- **Endpoint**: `GET /api/v1/user/credits` (Fetch current balance)
- **Endpoint**: `POST /api/v1/payments/unlock-fragment`
- **Frontend Trigger**: Decryption button click in the Decryption Gateway.
- **Logic**: Backend must verify credit balance before returning the `decrypted_content` (Markdown or JSON).

## 4. Memory Archive (Destiny Data Bank)
The Archive (`SCREEN_6`, `SCREEN_14`, `SCREEN_32`) displays historical data and allows timeline scrubbing.

- **Endpoint**: `GET /api/v1/archive/timeline`
- **Query Params**: `?timestamp={unix_time}`
- **Redacted Strata Logic**: The backend should return "corrupted" data blobs for failed paths, which the frontend renders with the glitch/redacted CSS classes defined in the designs.

## 5. State Persistence (Reality Sync)
The System Profile (`SCREEN_28`) summarizes the global state.

- **Endpoint**: `GET /api/v1/user/profile`
- **Key Metrics**:
  - `reality_sync_percentage`: Float (0.0 - 1.0)
  - `fragments_collected`: Integer
  - `strata_depth`: Integer

## 6. Frontend Hooks Integration
To connect the exported HTML/JS:
1. **Global State**: Use a store (e.g., Pinia, Redux, or a simple window object) to hold the `User_Token` and `Current_Sync_Status`.
2. **Dynamic Routes**: Replace static `<a href="...">` with a router-based approach that fetches narrative content on transition.
3. **Shader Triggers**: Call the `triggerGlitch()` or `initShader()` functions provided in the `<script>` blocks during API fetch wait times (Loading states).
